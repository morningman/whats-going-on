## **Objective**

An AI Data Infrastructure for **Multimodal Data** and **Agentic Interaction**

## **Core Capabilities**

* Columnar JSON 

* Hybrid Search 

* Multimodal Data （lance）

* Semantic Layer & MCP 

* AI SQL

## **Scenarios**

### Scenario 1: 业务日志分析 —— “非结构化数据的结构化治理”

* **核心痛点**：传统日志方案（如 ELK）存储成本高，Schema-on-write 模式处理动态变化的业务日志极不灵活。

* **业务描述**：业务日志由于迭代快、属性杂，通常呈现高度的非结构化特征。SelectDB 原生支持 **Variant 数据类型**，无需预定义繁琐的表结构，即可实现日志数据的“零加工”极速入库。Variant 兼具 Schema-free 的灵活性与传统物理列存的高性能，支持对复杂嵌套 JSON 的秒级索引与聚合分析。这不仅大幅降低了日志清洗（ETL）的运维成本，更让开发人员能够实时追踪业务动态，在海量日志中快速定位异常。

***

### Scenario 2: AI 可观测性 (AI Observability) —— “全栈性能的‘听诊器’”

* **核心痛点**：LLM 应用响应慢、Token 成本高、链路追踪（Trace）复杂。

* **业务描述**：AI 应用的上线只是起点，长效的性能监控与成本优化才是核心挑战。SelectDB 为 **LLM Traces、Logs 及 Metrics** 提供了统一的高性能存储方案，深度集成 Langfuse、OpenTelemetry 等主流可观测性标准。通过对 Prompt 响应耗时、Token 消耗及 LLM 评分的实时聚合分析，企业可以建立全维度的 AI 运行看板。利用 SelectDB 的实时预计算能力，运维团队能秒级洞察响应质量劣化或成本异常，确保 AI 业务在透明、可控的轨道上运行。

***

### Scenario 3: RAG / 知识库底座 —— “语义与关键词的双重精准打击”

* **核心痛点**：纯向量检索容易丢失专有名词（如型号、人名），纯文本检索无法理解深层语境。

* **业务描述**：针对大语言模型普遍面临的“幻觉”挑战，SelectDB 打造了混合搜索（Hybrid Search）**的终极底座。它将**全文本搜索（Full-text Search）**的精准匹配与**向量搜索（Vector Search）的语义理解深度融合在单一引擎中。在 RAG 流程中，SelectDB 能够在一行 SQL 中实现“关键词+语义”的加权检索，大幅提升召回率（Recall）。这种双剑合璧的能力，确保了 AI 生成的内容既具备逻辑相关性，又有物理事实的精准支撑，彻底解决 RAG 落地过程中的“搜不到”与“找不准”的问题。

***

### Scenario 4: Agentic Analytics —— “让 AI 拥有数据指挥权”

* **核心痛点**：AI Agent 无法直接理解数据库字段，SQL 生成（Text-to-SQL）在复杂业务场景下容易出错。

* **业务描述**：在 Agent 时代，AI 不仅要“思考”，更要能“自主行动”。SelectDB 通过原生集成 **MCP（Model Context Protocol）** 协议并引入**语义层（Semantic Layer）**，充当了 AI 模型与物理数据表之间的“翻译”。语义层将枯涩的数据库 Schema 映射为直观的业务概念，使 AI Agent 能够以自然语言进行确定性、可验证的数据交互。这种协议级的打通，让自主 Agent 能够实时调用 SelectDB 的分析能力进行自动化决策，真正实现“数据驱动指令，AI 驱动业务”的闭环。

***

### Scenario 5: 非构化数据SQL分析—— “SQL 即 AI，让数据分析无界”

* **核心痛点**：非结构化数据分析依赖 Python 和昂贵的 AI 工程团队，难以平民化。

* **业务描述**：为了打破 AI 应用的高门槛，SelectDB 将 LLM 推理能力原生嵌入数据库内核。用户无需搭建复杂的 Python 流水线，只需通过**标准 SQL 语句**即可调用大模型执行文本分类、情感识别、摘要提取及知识关联。这种“计算下推”的架构设计，极大地简化了非结构化数据的处理链路，实现了 AI 算子与 SQL 算子的无缝组合。它让每一位 SQL 开发人员都能直接在万亿级数据集上进行语义探索，让分散的非结构化文本瞬间转化为可量化、可决策的结构化洞察。

***

### Scenario 6: 多模态数据集管理 —— “AI 训练的‘粮仓’管理员”

* **用户角色**：算法工程师、MLOps 工程师

* **核心目标**：保障训练 / 测试数据集的可检索、可分析与高效管理。

* **业务描述**：
  面向音视频、文本、标注、特征等多模态训练数据，SelectDB作为统一的数据集元数据与样本管理引擎，提供**数据集元信息、样本分布、标注统计、质量指标**的快速查询与分析能力，支撑不同模型、不同实验间的数据可比性。支持**基于条件的样本检索与筛选**，**支撑模型自训练与数据反馈闭环，提升模型研发效率与质量一致性。**
